# Commit Instructions

These instructions reflect this repository's actual commit history and enforced hooks.

## Required Format

- Use Conventional Commits: `<type>(<scope>): <subject>`.
- Scope is optional, but recommended for non-trivial changes.
- Use one of the allowed types from commitlint:
  - `ci`, `chore`, `docs`, `ticket`, `feat`, `fix`, `perf`, `refactor`, `revert`, `style`

Examples:

- `feat(cli): add health endpoints and graceful shutdown handler`
- `fix(provider): add openstreetmap null-safety checks`
- `perf(core): optimize URLSearchParams construction`
- `docs: sync documentation`

## Scope Guidance (Project-Specific)

Prefer these scopes when they fit:

- `core` for `packages/core`
- `cli` for `packages/cli`
- `provider` for geocoding provider behavior
- `cache`, `throttler`, `loadbalancer` for decorator-level changes
- `rfc` or `rfc-###` for RFC/process artifacts
- `release` is reserved for release automation commits

If a change spans multiple areas, either:

- use a combined scope (for example `core,cli`), or
- omit scope and keep a clear subject.

## Subject Line Rules

- Use imperative mood: `add`, `fix`, `remove`, `refactor`, `optimize`.
- Keep it concise and specific.
- Start with lowercase after the colon.
- Do not end with a period.
- Do not prefix the header with emoji (`✨`, `✅`, etc.) because commitlint validates the header
  format strictly.

Good:

- `fix(cache/throttler): propagate AbortSignal and avoid caching aborted requests`

Avoid:

- `✨ feat: add new provider`
- `fix: Fixed several things.`

## Type Selection

- `feat`: new user-facing behavior
- `fix`: bug fix or correctness issue
- `perf`: measurable performance improvement
- `refactor`: internal restructuring without behavior change
- `docs`: documentation-only change
- `chore`: maintenance/tooling/release housekeeping
- `style`: formatting/style-only changes
- `ci`: CI pipeline/workflow changes
- `revert`: explicit rollback of a prior commit
- `ticket`: ticket-tracking or process-specific changes

## Hooks and Quality Gates

This repo enforces the following on commit:

- `commit-msg`: `commitlint` validation
- `pre-commit`: `turbo lint`, `turbo test`, `turbo build`

Practical impact:

- Commits can take time due to full lint/test/build checks.
- Keep commits focused to make failures easier to diagnose.

## Release Commit Policy

- Release commits are generated by `release-it` and follow:
  - `chore(release): vX.Y.Z`
- Do not manually mimic release commits unless you are intentionally performing the release flow.

## Recommended Workflow

1. Stage only related files.
2. Choose type and scope using rules above.
3. Write a concise imperative subject.
4. Commit and let hooks run.
5. If hooks fail, fix issues and create a new commit message only if scope/type changed.
